# P2/P3 Backlog Decomposition Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement one milestone at a time. Do not combine all milestones in one branch.

**Goal:** Convert issues #6 and #7 from broad epics into ordered, independently testable milestones.

**Architecture:** Work from runtime boundaries outward: stabilize backend interfaces and trace schemas, then improve coverage/platform support, then documentation, then product features. Each milestone must produce useful software independently and may be reviewed or rejected without blocking later unrelated work.

**Tech Stack:** TypeScript, Python, Bash, Vitest, JSON Schema, Mermaid, local-only web UI where applicable.

## Global Constraints

- Core npm package must remain useful without optional MMAS, kernel, or integration extras.
- No feature may expose a public port by default.
- Local credentials and machine-specific paths remain outside Git and npm.
- Every new backend or model registry change requires observable behavior tests.
- Each milestone receives its own design spec, implementation plan, branch, verification, and release decision.

---

## Milestone A: Backend and routing boundaries

Covers #6 P2-02, P2-03, P2-04, and P2-05.

1. Define `BackendAdapter` with `detect`, `health`, `dispatch`, `supportsStreaming`, and `configHints`.
2. Move hardcoded model aliases into a versioned model registry with local override support.
3. Define and validate a routing trace JSON schema containing task, signals, selected workflow, confidence, reasons, warnings, fallback reason, and timestamps.
4. Make CLI and proxy fallback policies share one source of truth or explicitly model two named policies.
5. Add adapter, registry, schema, and fallback-chain tests before migration.

Exit criteria: adding a backend does not require edits to unrelated commands, and every trace validates in CI.

## Milestone B: Core and extras package boundary

Covers #6 P2-01 and P2-07.

1. Write ADRs for npm lanes, vendored agent-kernel, MMAS subprocess isolation, proxy design, and Remote Agent policy.
2. Measure which files are required by Lane 1, Lane 2, and Lane 3.
3. Keep one package initially, but enforce explicit directories and optional runtime detection before considering package splitting.
4. Add package smoke tests for core-only behavior and extras-present behavior.

Exit criteria: npm users can identify which commands work without clone/extras, and optional components fail with actionable messages.

## Milestone C: Coverage and platform confidence

Covers #6 P2-08 through P2-14.

1. Add threshold tests for `scoreTask`.
2. Add conversion tests for `anthropicToOpenAI` and `openAIToAnthropic`, including malformed tool JSON.
3. Add mock streaming backend integration tests for Anthropic-compatible SSE output.
4. Snapshot strict no-network installer dry-run behavior.
5. Extend MMAS fake backend coverage for spawn, status, stop, and report.
6. Add WSL path assumptions to CI documentation and a path-normalization test script.

Exit criteria: critical conversion and process orchestration paths have deterministic tests without live LLM calls.

## Milestone D: Documentation truthfulness

Covers #6 P2-06 and #7 P2-15 through P2-20.

1. Archive historical audits or add an explicit historical banner.
2. Document every filesystem write and network call by lane, including rollback/uninstall.
3. Add command compatibility and troubleshooting matrices.
4. Add Mermaid diagrams for CLI dispatch, router, proxy fallback, and MMAS lifecycle.
5. Replace hardcoded marketing counts with generated or test-backed statistics.
6. Promote the strict dry-run install path before real installation.

Exit criteria: a new user can predict what each command touches and distinguish current architecture from historical audits.

## Milestone E: Product features

Covers #7 P3-01 through P3-10. Implement in this order:

1. `dt audit` because it protects all later automation.
2. `dt init project` and policy/ignore scaffolding.
3. `dt backends list` using the Milestone A adapter interface.
4. `dt traces list/show` using the trace schema.
5. `dt proxy test` with explicit provider opt-in and redacted output.
6. Policy packs: `strict`, `agency`, `oss-maintainer`, `enterprise`.
7. Backend telemetry with local-only storage and configurable retention.
8. Command recipes generated from verified workflows.
9. Local MMAS dashboard bound to loopback with no unauthenticated write actions.
10. Keep `dt run --dry-run` as an existing capability and expand tests/docs rather than rebuilding it.

Exit criteria: every command has a stable JSON mode where automation benefits, deny-by-default permissions, and no credential exposure.

## GitHub tracking decision

- Keep issues #6 and #7 open as roadmap epics until their milestones are split into implementation-specific issues.
- Add a comment to each epic pointing to this plan and recording completed items.
- Create child issues only when a milestone is ready for design and execution, avoiding dozens of speculative open tasks.
